## 在线对战核心问题

### 问题1: GitHub Pages部署WebSocket连接错误

**描述**:
GitHub Pages部署环境下，前端没有使用正确的Railway WebSocket地址，显示连接错误。

**根本原因**:
前端代码中WebSocket地址判断逻辑不完整，未区分GitHub Pages环境。

**解决方案**:
在 `vite.config.ts` 中添加环境变量配置：
```typescript
define: {
  'import.meta.env.VITE_GITHUB_PAGES': process.env.GITHUB_PAGES
}
```

在前端代码中使用环境变量：
```javascript
const WS_SERVER = new URLSearchParams(window.location.search).get('ws') || 
  (import.meta.env.VITE_GITHUB_PAGES ? 'wss://gomoku-game-production.up.railway.app' : '/ws');
```

**涉及文件**:
- `frontend/vite.config.ts`
- `frontend/src/stores/websocket.ts`

### 问题2: 玩家B加入房间后无法进入游戏页面

**描述**:
玩家A创建房间后，玩家B能加入房间，但是无法进入棋盘页面。

**解决方案**:

```javascript
case 'room_joined':

    // ... 游戏初始化
    currentScreen.value = 'game'; // 添加这一行
  } else {
    panels.value.waiting = true;
  }
```

### 问题4: 双方都显示自己是白棋

**描述**:
进入房间后，偶尔会出现玩家双方视角都显示自己是白棋、对方是黑棋。

**根本原因**:
1. 页面刷新时 `myColor` 默认值为 `1`（黑棋），导致加入者刷新后显示错误
2. 重连时 `rejoined` 消息没有恢复 `myColor`

**解决方案**:
1. 在 `savedRoomInfo` 中添加 `myColor` 字段
2. 在 `saveRoomInfo` 时保存 `myColor`
3. 在 `onMounted` 时从 localStorage 恢复 `myColor`
4. 在 `rejoined` 消息处理中恢复 `myColor`

```javascript
// 保存房间信息
function saveRoomInfo(roomCode, myName, myUserId, isHost, myColor) {
  savedRoomInfo.value = { roomCode, myName, myUserId, isHost, myColor };
  localStorage.setItem('gomoku-room', JSON.stringify({ 
    roomCode, myName, myUserId, isHost, myColor 
  }));
}

// 恢复房间信息
const savedRoom = wsStore.loadRoomInfo();
if (savedRoom) {
  gameStore.myColor = savedRoom.myColor;
  gameStore.isHost = savedRoom.isHost;
}

// 重连恢复
case 'rejoined':
  if (data.color !== undefined) {
    gameStore.myColor = data.color;
  }
```

### 问题5: 房主看不到比分

**描述**:
在在线对战中，房主视角看不见比分，非房主玩家能看到。

**根本原因**:
`opponent_joined` 消息中缺少 `matchMode` 字段，导致房主的 `matchMode` 被设置为默认值 `1`（单局模式），比分显示条件 `matchMode > 1` 不成立。

**解决方案**:
在 `opponent_joined` 消息中添加配置信息：
```javascript
send(room.players[0].ws, { 
  type: 'opponent_joined', 
  opponent: room.players[1],
  gameTime: room.gameTime,
  boardSize: room.boardSize,
  matchMode: room.matchMode, // 添加这一行
  undoLimit: room.undoLimit
});
```

### 问题6: 悔棋次数没有扣减

**描述**:
在线对战悔棋时，偶尔出现悔棋次数没有扣减的情况。

**根本原因**:
1. 后端 `handleUndoAccept` 函数中执行了 `moves--` 但忘记扣减 `undoLeft`
2. 前端在 `undo_accepted` 消息处理中自己扣减 `undoLeft`，但扣减的是接收消息方而不是发起悔棋请求方

**解决方案**:
1. 后端在 `handleUndoAccept` 中添加 `undoLeft--` 并发送最新状态
2. 前端从消息中同步 `undoLeft` 值

```javascript
// 后端
function handleUndoAccept(ws) {
  // ... 悔棋逻辑
  room.players[lastPlayerIndex].undoLeft--; // 扣减悔棋次数
  
  broadcastToRoom(roomCode, { 
    type: 'undo_accepted',
    players: room.players.map(p => p ? {
      time: p.time,
      moves: p.moves,
      undoLeft: p.undoLeft, // 发送最新悔棋次数
      color: p.color
    } : null)
  });
}

// 前端
case 'undo_accepted':
  gameStore.doUndo();
  if (data.players) {
    data.players.forEach((p: any) => {
      if (p) {
        gameStore.players[p.color].undoLeft = p.undoLeft; // 同步悔棋次数
      }
    });
  }
  showToast('悔棋成功');
```

**涉及文件**:
- `server/server.js`
- `frontend/src/App.vue`

### 问题7: 倒计时不同步

**描述**:
玩家A显示自己剩余3分钟，玩家B显示A剩余4分钟，时间不一致。

**根本原因**:
- 原设计由客户端各自维护倒计时，由于网络延迟和计算时间差异导致不同步
- 断线重连后没有正确恢复时间

**解决方案**:
1. 服务器每秒自动扣减当前玩家时间
2. 服务器每秒发送 `time_sync` 消息同步双方时间
3. 前端接收并更新显示

```javascript
// 服务器端
setInterval(() => {
  rooms.forEach((room, roomCode) => {
    if (room.status === 'playing') {
      const currentPlayerIndex = room.players.findIndex(p => p && p.color === room.currentPlayer);
      if (currentPlayerIndex !== -1) {
        room.players[currentPlayerIndex].time--;
        
        // 同步时间给所有客户端
        broadcastToRoom(roomCode, {
          type: 'time_sync',
          players: room.players.map(p => p ? {
            time: p.time,
            color: p.color
          } : null),
          currentPlayer: room.currentPlayer
        });
      }
    }
  });
}, 1000);

// 前端
case 'time_sync':
  if (data.players) {
    data.players.forEach((p: any) => {
      if (p) {
        gameStore.players[p.color].time = p.time;
      }
    });
  }
  if (data.currentPlayer) {
    gameStore.currentPlayer = data.currentPlayer;
  }
```

### 问题8: 悔棋逻辑错误

**描述**:
在线对战悔棋逻辑不正确，应该在自己落子后、对方落子前才可以发起悔棋。

**根本原因**:
原悔棋按钮启用条件过于宽松，允许任何时候悔棋。

**解决方案**:
修改悔棋按钮启用条件，只允许在自己落子后、对方落子前悔棋：

```javascript
const canUndo = computed(() => {
  // ... 其他模式判断
  
  if (props.gameState.gameMode === 'online') {
    const isMyTurn = currentPlayer === myColor;
    const lastMove = props.gameState.moveHistory[props.gameState.moveHistory.length - 1];
    const isMyLastMove = lastMove && lastMove.player === myColor;
    
    // 只能在自己落子后、对方落子前悔棋
    return isMyTurn && // 轮到对方
           isMyLastMove && // 最后一手是自己下的
           props.gameState.players[myColor].undoLeft > 0 &&
           props.gameState.moveHistory.length > 0;
  }
  
  return false;
});
```

**涉及文件**:
- `frontend/src/components/game/GameControls.vue`

## 数据一致性问题

### 问题22: 前后端状态不一致

**描述**:
前后端状态管理方式不同，容易导致数据不一致。

**根本原因**:
- 后端使用位置索引：`players[0]`（房主）、`players[1]`（加入者）
- 前端使用颜色索引：`players[1]`（黑棋）、`players[2]`（白棋）

**解决方案**:
1. 统一使用 `color` 字段进行映射
2. 所有状态更新由服务器发起
3. 前端只负责显示，不进行状态计算

**核心原则**:
```javascript
// 后端
room.players[playerIndex].color = 1; // 或 2

// 前端
gameStore.players[p.color] = p; // p.color 是 1 或 2
```

**涉及文件**:
- `server/server.js`
- `frontend/src/App.vue`
- `frontend/src/stores/game.ts`

### 问题10: Docker部署WebSocket消息格式错误

**描述**:
Docker部署环境下，通过IP访问时，玩家加入房间显示"消息格式错误"，但玩家能进入房间并发送消息，房主无法进入页面但能收到消息。

**解决方案**:
只发送必要的属性，避免包含 `ws`：
```javascript
send(room.players[0].ws, {
  type: 'opponent_joined',
  opponent: room.players[1] ? {
    name: room.players[1].name,
    time: room.players[1].time,
    moves: room.players[1].moves,
    undoLeft: room.players[1].undoLeft,
    color: room.players[1].color
  } : null,
});
```

**影响范围**:
- `handleJoinRoom` 函数（两处）
- 参考 `handleRejoinRoom` 的正确实现

**验证方法**:
查看服务器日志中的异常信息：
```
TypeError: Converting circular structure to JSON
at JSON.stringify
at send
at handleJoinRoom
```

### 问题13: 房间销毁逻辑不一致

**描述**:

**解决方案**:
```javascript

  const isHost = userId === room.hostId;
  
  if (isHost) {
    // 房主离开，删除房间
    rooms.delete(roomCode);
    broadcastToRoom(roomCode, { type: 'opponent_left' });
  } else {
    // 加入者离开，只通知房主
    const host = room.players.find(p => p && p.id === room.hostId);
    if (host && host.ws) {
      send(host.ws, { type: 'opponent_left' });
    }
  }
}
```

### 问题20: 复制房间链接兼容性问题

**描述**:
Docker部署后通过IP访问时，复制房间链接失败，显示"消息格式错误"。

**根本原因**:
Clipboard API 在非安全上下文（HTTP + 非localhost）中被禁用。

**解决方案**:
```javascript
async function copyLink() {
  const url = getRoomUrl();
  
  try {
    // 方案 A: 现代 API (HTTPS 环境)
    if (navigator.clipboard && window.isSecureContext) {
      await navigator.clipboard.writeText(url);
      copySuccess.value = true;
      return;
    }

    // 方案 B: 降级方案（创建临时 textarea）
    const textArea = document.createElement('textarea');
    textArea.value = url;
    textArea.style.position = 'fixed';
    textArea.style.left = '-9999px';
    textArea.style.top = '-9999px';
    textArea.style.opacity = '0';
    textArea.setAttribute('readonly', '');
    
    document.body.appendChild(textArea);
    textArea.focus();
    textArea.select();
    
    const success = document.execCommand('copy');
    document.body.removeChild(textArea);
    
    if (success) {
      copySuccess.value = true;
    }
  } catch (error) {
    console.error('复制失败:', error);
  }
}
```

**关键要点**:
- 先检查条件，避免访问 `undefined` 对象
- 降级方案作为正常流程，不在 `.catch()` 中
- 检查 `execCommand` 返回值
